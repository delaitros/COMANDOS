CAPÍTULO 00

¿Qué es una red informática?

Antes de hablar de comandos, hay que entender sobre qué trabajan.

Una red informática es un conjunto de dispositivos (computadoras, routers, teléfonos, servidores) interconectados que pueden intercambiar datos entre sí. Esos datos viajan en forma de pequeños bloques llamados paquetes.

Pensá en la red como una ciudad: cada dispositivo es una casa con su propia dirección (IP), las calles son los cables o el WiFi, y los routers son las intersecciones que deciden por dónde manda el tráfico.

DATO INTERESANTE

Internet no es "la nube" ni un lugar abstracto. Es una red de redes físicas conectadas entre sí: millones de kilómetros de cables de fibra óptica bajo tierra y bajo el océano, más satélites. Los cables submarinos transportan el 95% del tráfico internacional de internet.

¿Cómo viajan los datos?

Los datos se dividen en paquetes antes de enviarse. Cada paquete tiene una cabecera (header) con información de origen, destino y número de secuencia, y un cuerpo (payload) con los datos reales. Los paquetes pueden tomar distintas rutas y se reensamblan al llegar.

Tu PC Router 1 Router 2 Servidor [192.168.1.100] ──────► [10.0.0.1] ──────► [200.51.12.1] ──────► [142.250.x.x] Paquete #1 ──────────────────────────────────────────────────────────────────────────► Paquete #2 ─────────────────────────────────────────────────────────────────────► Paquete #3 (ruta alternativa) ──────────────────────────────────────────────────────► Al llegar: se reensamblan en orden usando los números de secuencia
CAPÍTULO 01

Modelos OSI y TCP/IP

El mapa conceptual de toda comunicación en red.

Para entender cómo se comunican los dispositivos, existen dos modelos de referencia que dividen el proceso en capas. Cada capa tiene una responsabilidad específica y usa los servicios de la capa inferior.

Modelo OSI — 7 capas

El modelo OSI (Open Systems Interconnection) divide la comunicación en 7 capas. Es un modelo teórico que sirve para entender y diagnosticar problemas.

7 — APLICACIÓNHTTP, FTP, DNS, SMTP
6 — PRESENTACIÓNSSL/TLS, compresión
5 — SESIÓNNetBIOS, RPC
4 — TRANSPORTETCP, UDP
3 — REDIP, ICMP, ARP
2 — ENLACE DE DATOSEthernet, WiFi, MAC
1 — FÍSICACable, fibra, radio
MNEMOTECNIA

Para recordar las 7 capas de abajo hacia arriba: "Por Favor Dame Tres Sillas Perfectas Ahora" → Física, Enlace, Red, Transporte, Sesión, Presentación, Aplicación.

Modelo TCP/IP — 4 capas (el real)

El modelo TCP/IP es el que realmente usa Internet. Simplifica las 7 capas de OSI en 4:

Capa TCP/IPEquivale a OSIProtocolos
Aplicación5, 6, 7HTTP, DNS, FTP, SMTP, SSH
Transporte4TCP, UDP
Internet3IP, ICMP, ARP
Acceso a red1, 2Ethernet, WiFi, PPP

Cuando ejecutás ping, trabajás en la capa de Internet (capa 3) usando el protocolo ICMP. Cuando navegás a un sitio web, usás HTTP/HTTPS en la capa de Aplicación sobre TCP en la capa de Transporte.

COMANDO 01

PING WIN + LINUX

El comando más fundamental del diagnóstico de red.

PING (Packet Internet Groper) es el primer comando que usa cualquier administrador de red cuando hay un problema. Responde la pregunta más básica: "¿Este dispositivo está vivo y puede comunicarse conmigo?"

Cómo funciona

Ping envía pequeños paquetes al destino y espera que los devuelva. Midiendo cuánto tarda ese proceso, podemos saber:

C:\> ping 8.8.8.8 Haciendo ping a 8.8.8.8 con 32 bytes de datos: Respuesta desde 8.8.8.8: bytes=32 tiempo=18ms TTL=118 ← respuesta recibida en 18ms Respuesta desde 8.8.8.8: bytes=32 tiempo=20ms TTL=118 Respuesta desde 8.8.8.8: bytes=32 tiempo=17ms TTL=118 Tiempo de espera agotado para esta solicitud. ← 1 paquete se perdió Estadísticas de ping para 8.8.8.8: Paquetes: enviados = 4, recibidos = 3, perdidos = 1 (25% perdidos) Tiempos aproximados de ida y vuelta en milisegundos: Mínimo = 17ms, Máximo = 20ms, Media = 18ms

Flags más importantes

-n [num] (Win)
-c [num] (Linux)Número de paquetes a enviar. Por defecto Windows envía 4, Linux envía infinitos hasta Ctrl+C.
-t (Win)
(sin flag en Linux)Ping continuo hasta que el usuario lo detenga con Ctrl+C. Útil para monitorear en tiempo real.
-l [bytes] (Win)
-s [bytes] (Linux)Tamaño del paquete. Por defecto 32 bytes en Windows, 56 en Linux. Máximo ~65500 bytes.
-i [ttl] (Win)
-t [ttl] (Linux)Establecer manualmente el TTL del paquete saliente.
-4 / -6Forzar uso de IPv4 o IPv6.
IMPORTANTE

Que un host no responda al ping no significa que esté apagado. Muchos firewalls y servidores bloquean ICMP por seguridad. Por ejemplo, ping microsoft.com suele fallar aunque el sitio esté en línea.

Sub-concepto: Protocolo ICMP

ICMP (Internet Control Message Protocol, RFC 792) es el protocolo de mensajería de la red. No transporta datos de usuario sino mensajes de control y error entre dispositivos de red.

Opera en la Capa 3 (Red) del modelo OSI, junto con IP. Los paquetes ICMP van encapsulados dentro de paquetes IP pero no usan puertos TCP/UDP.

Tipo ICMPNombreDescripción
0Echo ReplyRespuesta al ping
3Destination UnreachableEl destino no es alcanzable
5RedirectUsar otra ruta
8Echo RequestEl ping en sí
11Time ExceededTTL llegó a 0 (usado por tracert)
DATO CURIOSO

El nombre PING fue inspirado en el sonar de los submarinos: emiten un pulso sónico ("ping") y escuchan el eco para detectar objetos. El creador del comando, Mike Muuss, lo nombró así en 1983 cuando tenía solo 15 años. Murió en un accidente de tráfico en el año 2000.

Sub-concepto: TTL (Time To Live)

TTL es un campo en la cabecera IP que limita la vida de un paquete. Es un contador que cada router decrementa en 1 al reenviar el paquete. Cuando llega a 0, el router descarta el paquete y envía un mensaje ICMP "Time Exceeded" al origen.

Su propósito es evitar que los paquetes circulen infinitamente por la red si hay un loop de rutas.

Tu PC (TTL=128) │ ▼ Router 1 → TTL ahora = 127 │ ▼ Router 2 → TTL ahora = 126 │ ▼ Router 3 → TTL ahora = 125 │ ▼ ... y así hasta llegar al destino o hasta TTL=0 Si TTL llega a 0: el router descarta el paquete y te avisa con ICMP Type 11
Sistema OperativoTTL por defecto
Windows128
Linux / Android64
macOS / iOS64
Cisco (routers)255
DATO ÚTIL

Cuando hacés ping a una IP y la respuesta tiene TTL=118, podés deducir que el servidor usa Windows (porque 128 - 118 = 10 saltos hasta llegar a vos). Si el TTL es 54, es probablemente Linux (64 - 54 = 10 saltos).

Sub-concepto: RTT y Latencia

RTT (Round Trip Time) es el tiempo que tarda un paquete en ir al destino y volver. Es lo que ping muestra en milisegundos (ms). También se llama latencia.

LatenciaContexto típicoCalidad
< 5 msRed local (LAN)Excelente
5 - 30 msCiudad o país cercanoMuy buena
30 - 100 msPaís distanteAceptable
100 - 200 msOtro continentePerceptible
> 200 msMuy lejos o red saturadaMala

Jitter es la variación en la latencia: si un ping da 18ms, 19ms, 80ms, 17ms, ese 80ms es jitter. Es especialmente malo para videollamadas y juegos online.

COMANDO 02

TRACERT / TRACEROUTE WIN + LINUX

Descubrí el camino exacto que toman tus datos.

TRACERT en Windows y TRACEROUTE en Linux trazan la ruta completa que sigue un paquete desde tu computadora hasta el destino, mostrando cada router intermedio con su latencia.

El truco del TTL incremental

La técnica es ingeniosa: se envían paquetes con TTL=1, 2, 3, 4... sucesivamente. El primer router recibe el paquete con TTL=1, lo decrementa a 0, lo descarta y te envía un ICMP "Time Exceeded" — revelando así su IP. El segundo router hace lo mismo con TTL=2. Así se van descubriendo todos los routers del camino.

ENVÍO TTL=1: Tu PC ──► Router1 (TTL→0) ──✗── "Time Exceeded" ──► Tu PC descubre Router1 ENVÍO TTL=2: Tu PC ──► Router1 ──► Router2 (TTL→0) ──✗── "Time Exceeded" ──► Router2 ENVÍO TTL=3: Tu PC ──► Router1 ──► Router2 ──► Router3 (TTL→0) ──✗── Router3 ...y así hasta llegar al destino final
C:\> tracert google.com Traza la ruta a google.com [142.250.184.78] mediante 30 saltos como máximo: 1 2 ms 2 ms 2 ms 192.168.1.1 ← tu router hogareño 2 8 ms 9 ms 8 ms 10.10.0.1 ← router de tu ISP 3 15 ms 14 ms 16 ms 200.51.12.1 ← backbone del ISP 4 22 ms 21 ms 23 ms 72.14.215.1 ← red de Google (Peering) 5 * * * Tiempo de espera de la solicitud agotado. ← firewall 6 25 ms 24 ms 25 ms 142.250.184.78 ← servidor destino Traza completa.

Sub-concepto: Saltos (Hops) y Routers

Cada "salto" o hop es un router que reenvía el paquete hacia el destino. Los routers son dispositivos especializados que leen la dirección IP destino de cada paquete y deciden por qué interfaz reenviarlo, usando su tabla de enrutamiento.

Los asteriscos (*) en tracert no significan que el router esté caído: simplemente tiene configurado un firewall que bloquea los paquetes ICMP con TTL expirado. El tráfico real sí pasa por él.

DATO INTERESANTE

La diferencia técnica entre Windows y Linux: tracert (Windows) usa paquetes ICMP Echo Request. traceroute (Linux) usa paquetes UDP por defecto (a puertos altos 33434+). Por eso en Linux se puede usar traceroute -I para que use ICMP como Windows. Esto importa porque algunos firewalls tratan diferente a ICMP y UDP.

COMANDO 03

IPCONFIG / IFCONFIG / IP ADDR WIN + LINUX

Tu DNI en la red: todo sobre la identidad de tu dispositivo.

Estos comandos muestran (y permiten modificar) la configuración de red de todas las interfaces del sistema. Son el punto de partida para cualquier diagnóstico.

C:\> ipconfig /all Adaptador Ethernet Ethernet: Descripción: Realtek PCIe GbE Family Controller Dirección física: A4-B1-C7-3F-22-09 ← MAC Address DHCP habilitado: Sí ← IP asignada automáticamente Dirección IPv4: 192.168.1.100 ← tu IP en la red local Máscara de subred: 255.255.255.0 ← define el tamaño de la red Puerta de enlace predeterminada: 192.168.1.1 ← el router Servidores DNS: 8.8.8.8, 8.8.4.4 ← traductores de nombres

Sub-concepto: Dirección IP

Una dirección IP (Internet Protocol) es el identificador único de un dispositivo en una red. Funciona como la dirección postal: sin ella, nadie sabe cómo enviarte datos.

IPv4

IPv4 usa 32 bits divididos en 4 grupos de 8 bits (octetos), escritos en decimal separados por puntos: 192.168.1.100. Permite ~4.300 millones de direcciones únicas.

PROBLEMA ACTUAL

Las direcciones IPv4 ya se agotaron en 2011. Por eso hoy se usa NAT (Network Address Translation): múltiples dispositivos comparten una sola IP pública usando IPs privadas internamente.

Rangos de IP privadas (RFC 1918)

RangoUso típicoCantidad de IPs
10.0.0.0 – 10.255.255.255Empresas grandes16 millones
172.16.0.0 – 172.31.255.255Empresas medianas1 millón
192.168.0.0 – 192.168.255.255Hogares y oficinas65.536
127.0.0.0 – 127.255.255.255Loopback (el mismo equipo)16 millones

IPv6

IPv6 usa 128 bits escritos en hexadecimal: 2001:0db8:85a3:0000:0000:8a2e:0370:7334. Permite 340 undecillones de direcciones (340 seguido de 36 ceros). Fue diseñado para reemplazar IPv4, pero la transición es lenta.

Sub-concepto: Máscara de subred

La máscara de subred define qué parte de una dirección IP identifica la red y qué parte identifica al host dentro de esa red.

IP: 192.168.1 . 100 Máscara: 255.255.255 . 0 [──────────────] [───] RED HOST Todos los dispositivos con 192.168.1.X están en la misma red. El .100 identifica a este dispositivo específico. En notación CIDR: 192.168.1.0/24 (24 bits para la red, 8 para host → 254 hosts)

La notación CIDR (ej: /24) indica cuántos bits son para la red. Una red /24 tiene 256 IPs (254 usables), una /16 tiene 65.536.

Sub-concepto: Gateway (Puerta de enlace)

El gateway predeterminado es la IP del router de tu red local. Cuando querés comunicarte con una IP que no está en tu red local (como 142.250.x.x de Google), tu PC envía el paquete al gateway, que sabe cómo enviarlo hacia internet.

¿Destino 192.168.1.50? → Está en mi red → Envío directo (ARP) ¿Destino 8.8.8.8? → NO está en mi red → Lo envío a mi GATEWAY (192.168.1.1)

Sub-concepto: DHCP

DHCP (Dynamic Host Configuration Protocol) es el protocolo que asigna automáticamente una IP a tu dispositivo cuando se conecta a la red. Sin DHCP, tendrías que configurar manualmente IP, máscara, gateway y DNS en cada dispositivo.

El proceso DHCP tiene 4 pasos (DORA):

1. DISCOVER → Tu PC grita: "¿Hay algún servidor DHCP?" 2. OFFER → El router responde: "Te ofrezco la IP 192.168.1.100" 3. REQUEST → Tu PC dice: "Acepto esa IP" 4. ACK → El router confirma: "Es tuya por 24 horas (lease time)"
PARA SABER

ipconfig /release le dice al DHCP "ya no uso más esta IP". ipconfig /renew inicia el proceso DORA de nuevo para obtener una IP (la misma u otra). Útil cuando la red no funciona bien.

Sub-concepto: Dirección MAC

La dirección MAC (Media Access Control) es el identificador físico y único de cada interfaz de red, grabado por el fabricante. Se usa para la comunicación dentro de la misma red local (Capa 2), antes de llegar a la capa IP.

Tiene 48 bits escritos en hexadecimal: A4:B1:C7:3F:22:09. Los primeros 3 bytes (A4:B1:C7) identifican al fabricante (OUI — Organizationally Unique Identifier).

DATO CURIOSO

Las MACs son teóricamente únicas a nivel mundial, pero pueden spoofarse (falsificarse) por software. Muchos sistemas operativos modernos usan MAC aleatoria al conectarse a redes WiFi para proteger la privacidad del usuario.

COMANDO 04

NETSTAT WIN + LINUX

El detector de conexiones sospechosas.

NETSTAT (Network Statistics) muestra en tiempo real todas las conexiones de red activas, los puertos que están escuchando y las estadísticas del stack de red. Es una herramienta esencial para detectar malware o conexiones no autorizadas.

C:\> netstat -ano Proto Dirección local Dirección remota Estado PID TCP 192.168.1.100:49823 142.250.184.78:443 ESTABLISHED 4892 ← Chrome en Google TCP 192.168.1.100:49891 140.82.114.4:443 ESTABLISHED 7234 ← GitHub TCP 0.0.0.0:22 0.0.0.0:0 LISTENING 1240 ← SSH escuchando TCP 0.0.0.0:80 0.0.0.0:0 LISTENING 980 ← servidor web TCP 192.168.1.100:52341 203.0.113.42:4444 ESTABLISHED 3891 ← ¡sospechoso!

Sub-concepto: Puertos de red

Los puertos son números del 0 al 65535 que permiten que un mismo dispositivo (con una sola IP) maneje múltiples servicios simultáneamente. Son como los "departamentos" dentro de un edificio (la IP es el edificio, el puerto es el departamento).

Sub-concepto: Estados TCP

TCP es un protocolo orientado a conexión: antes de enviar datos, establece un canal. Ese canal pasa por varios estados:

EstadoSignificado
LISTENINGPuerto abierto, esperando conexiones entrantes
SYN_SENTSe envió solicitud de conexión, esperando respuesta
SYN_RECEIVEDSe recibió solicitud, enviando confirmación
ESTABLISHEDConexión activa, datos fluyendo
FIN_WAIT_1/2Proceso de cierre de conexión
TIME_WAITEsperando para asegurarse que se recibió el cierre

SEGURIDAD

Un SYN Flood es un ataque DoS que envía miles de paquetes SYN sin completar el handshake, llenando la tabla de conexiones en estado SYN_RECEIVED hasta agotar los recursos del servidor.

Sub-concepto: TCP vs UDP

Son los dos protocolos de transporte principales. La diferencia clave: TCP garantiza la entrega, UDP no.

CaracterísticaTCPUDP
ConfiabilidadGarantizada (retransmite si hay pérdida)No garantizada
OrdenEntrega en ordenSin orden garantizado
VelocidadMás lento (overhead)Más rápido (mínimo overhead)

ANALOGÍA

TCP es como enviar una carta certificada: el cartero confirma la entrega y si no llegó, la re-envía.
UDP es como lanzar panfletos desde un avión: llegan rápido, pero no sabés cuántos llegaron ni en qué orden.

COMANDO 05

NSLOOKUP / DIG WIN + LINUX

El sistema telefónico de Internet: DNS.

NSLOOKUP (Name Server Lookup) y DIG (Domain Information Groper) consultan el sistema DNS para resolver nombres de dominio. DIG es más potente y detallado, preferido en Linux para diagnósticos avanzados.

$ dig google.com ; <<>> DiG 9.18.1 <<>> google.com ;; ANSWER SECTION: google.com. 300 IN A 142.250.184.78 ← IP de Google ;; Query time: 12 msec ← tardó 12ms en resolver ;; SERVER: 8.8.8.8#53(8.8.8.8) ← usó el DNS de Google para resolver $ dig gmail.com MX ;; ANSWER SECTION: gmail.com. 3600 IN MX 10 aspmx.l.google.com. gmail.com. 3600 IN MX 20 alt1.aspmx.l.google.com.

Sub-concepto: Cómo funciona DNS

DNS (Domain Name System) es la "guía telefónica" de Internet. Traduce nombres legibles como google.com a direcciones IP que las computadoras entienden como 142.250.184.78.

Es un sistema jerárquico y distribuido: no hay un único servidor que sabe todo, sino una cadena de servidores que se consultan entre sí.

Escribís: www.google.com 1. Tu PC pregunta a su DNS Resolver (8.8.8.8 o el del router) ↓ 2. El Resolver pregunta al Root Server (.) "¿Quién sabe sobre .com?" ↓ 3. Root Server responde: "Preguntale a los TLD Servers del .com" ↓ 4. Resolver pregunta al TLD Server del .com "¿Quién sabe sobre google.com?" ↓ 5. TLD Server responde: "Preguntale al Authoritative Server de Google" ↓ 6. Resolver pregunta al servidor autoritativo de Google "¿Cuál es la IP de www.google.com?" ↓ 7. Google responde: "Es 142.250.184.78" ↓ 8. El Resolver te responde y guarda en caché por el tiempo del TTL
DATO INTERESANTE

Solo hay 13 grupos de Root Servers (A hasta M), pero cada grupo tiene cientos de servidores físicos distribuidos por todo el mundo via Anycast. Son los servidores más críticos de Internet: si todos cayeran, nadie podría resolver nombres de dominio.

DNS CACHE POISONING

Un ataque donde un atacante logra que el resolver DNS guarde en caché una respuesta falsa. Así cuando escribís banco.com, tu DNS resuelve a una IP maliciosa en vez de la real. Por eso existe DNSSEC (DNS Security Extensions).

COMANDO 06

ARP WIN + LINUX

El puente entre la dirección IP y la dirección física.

ARP (Address Resolution Protocol, RFC 826) resuelve el problema de: "Sé la IP del dispositivo, pero ¿cuál es su dirección MAC para poder enviarle el frame Ethernet?"

Sub-concepto: Cómo funciona ARP

Cuando tu PC quiere enviar un paquete a una IP en su misma red local, necesita la MAC de ese destino para armar el frame de Ethernet. Si no la tiene en caché, ejecuta una consulta ARP:

Tu PC (192.168.1.100) quiere hablar con 192.168.1.50 ARP REQUEST (broadcast): "Hola a TODOS en la red: ¿Quién tiene la IP 192.168.1.50? Soy 192.168.1.100 con MAC A4:B1:C7:3F:22:09" ↓ (va a FF:FF:FF:FF:FF:FF — todos escuchan) ARP REPLY (unicast): "¡Yo! Soy 192.168.1.50 y mi MAC es DC:A6:32:9F:AA:11" ↓ Tu PC guarda en la tabla ARP: 192.168.1.50 → DC:A6:32:9F:AA:11 (Esta entrada dura ~2 minutos antes de expirar)
C:\> arp -a Interfaz: 192.168.1.100 --- 0x4 Dirección de Internet Dirección física Tipo 192.168.1.1 a4-b1-c7-3f-22-01 dinámico ← router 192.168.1.102 dc-a6-32-9f-aa-11 dinámico ← otro PC 192.168.1.255 ff-ff-ff-ff-ff-ff estático ← broadcast

Sub-concepto: ARP Spoofing / ARP Poisoning

El protocolo ARP tiene un problema de seguridad fundamental: no tiene autenticación. Cualquier dispositivo puede enviar una respuesta ARP no solicitada, y los demás la aceptarán y actualizarán su tabla.

Un atacante puede explotar esto con ARP Spoofing: enviar respuestas ARP falsas asociando su propia MAC a la IP del gateway, haciendo que todo el tráfico de la víctima pase por el atacante primero.

SIN ATAQUE: Víctima → Gateway (router) → Internet CON ARP SPOOFING (Man-in-the-Middle): Atacante envía ARP falso a la víctima: "La MAC del router (192.168.1.1) es XX:XX:XX:XX:XX:XX (mi MAC)" Resultado: Víctima → Atacante → Gateway → Internet ↑ El atacante lee/modifica todo el tráfico
DEFENSA

Dynamic ARP Inspection (DAI) en switches administrados valida las respuestas ARP contra una tabla DHCP confiable. También se puede usar ARP estático para el gateway, o protocolos como 802.1X. En redes domésticas, usar HTTPS cifra el contenido aunque el tráfico sea interceptado.

COMANDO 08

NMAP WIN + LINUX

La navaja suiza del reconocimiento de redes.

NMAP (Network Mapper) es la herramienta de escaneo de redes más utilizada en el mundo. Creada por Gordon Lyon ("Fyodor") en 1997, sigue siendo el estándar de la industria para reconocimiento de redes.

ÉTICO Y LEGAL

Escanear redes o sistemas sin autorización explícita es ilegal en Argentina (Ley 26.388) y en la mayoría de los países. NMAP debe usarse únicamente en redes propias, en laboratorios controlados, o en auditorías con contrato firmado.

Sub-concepto: TCP Three-Way Handshake

Antes de entender los tipos de escaneo, hay que entender cómo TCP establece una conexión: el famoso 3-way handshake (apretón de manos de tres pasos).

Cliente Servidor │ │ │──────── SYN ──────────────────────►│ "Quiero conectarme" │ │ │◄─────── SYN-ACK ──────────────────│ "Ok, te escucho. Confirmás?" │ │ │──────── ACK ──────────────────────►│ "Confirmado. Conexión ESTABLISHED" │ │ │◄══════ DATOS FLUYEN ══════════════►│

Los flags TCP son bits de control en la cabecera TCP:

SYNSynchronize — inicia una conexión
ACKAcknowledge — confirma recepción
FINFinish — cierra una conexión gracefully
RSTReset — cierra una conexión abruptamente (puerto cerrado)
PSHPush — enviar datos inmediatamente al buffer de la app
URGUrgent — datos urgentes

Sub-concepto: Tipos de escaneo de puertos

SYN Scan (-sS) — el más usado

También llamado "half-open scan" o "stealth scan". Envía un SYN, si recibe SYN-ACK el puerto está abierto, y en lugar de completar el handshake envía un RST para cerrar sin establecer conexión completa. Es más sigiloso porque no queda registrado en muchos logs.

Puerto ABIERTO: → SYN → ← SYN-ACK ← → RST → (no completa conexión) Puerto CERRADO: → SYN → ← RST ← (respuesta inmediata) Puerto FILTRADO: → SYN → (no hay respuesta, firewall descarta silenciosamente)

TCP Connect Scan (-sT)

Completa el handshake completo. Más detectable (queda en logs), pero funciona sin privilegios de root.

UDP Scan (-sU)

Escanea puertos UDP enviando paquetes vacíos. Si recibe ICMP "Port Unreachable", el puerto está cerrado. Sin respuesta, puede estar abierto o filtrado. Es muy lento.

Version Detection (-sV)

Una vez identificados los puertos abiertos, envía probes específicos para identificar el servicio y su versión exacta. Ej: "Apache httpd 2.4.52", "OpenSSH 8.9p1".

OS Detection (-O)

Analiza pequeñas diferencias en cómo el SO responde a ciertos paquetes (TCP/IP fingerprinting) para identificar el sistema operativo. Requiere al menos un puerto abierto y uno cerrado.

DATO CULTURAL

NMAP apareció en varias películas: en Matrix Reloaded (2003), Trinity usa NMAP 2.54BETA25 para escanear una central eléctrica. En The Dark Knight Rises y otras películas también se ven pantallas con salidas de NMAP. Los directores las usan porque parecen "reales" (porque lo son).

COMANDO 09

PATHPING / MTR WIN + LINUX

Estadísticas detalladas por cada salto de la ruta.

PATHPING (Windows) y MTR (My Traceroute, Linux) son la evolución natural de tracert. En vez de solo mostrar los saltos una vez, envían múltiples paquetes a cada router y calculan estadísticas de pérdida y latencia para cada hop individualmente.

Esto permite identificar con precisión en qué router específico está ocurriendo la pérdida de paquetes o la latencia elevada. PATHPING primero traza la ruta (como tracert), luego envía 100 paquetes a cada salto y calcula las estadísticas.

C:\> pathping google.com [Primero traza la ruta...] 0 MI-PC [192.168.1.100] 1 192.168.1.1 2 200.51.12.1 3 72.14.215.1 4 142.250.184.78 Calculando estadísticas durante 125 segundos... Origen a aquí Este nodo/vínculo Salto RTT Perdido/Env Perdido/Env Dirección 1 2ms 0/100=0% 0/100=0% 192.168.1.1 2 15ms 0/100=0% 0/100=0% 200.51.12.1 3 * 3/100=3% 3/100=3% 72.14.215.1 ← 3% de pérdida aquí 4 25ms 3/100=3% 0/100=0% 142.250.184.78

MTR en Linux hace lo mismo pero en tiempo real, actualizando las estadísticas constantemente mientras corre.

COMANDO 10

NETSH / NMCLI WIN + LINUX

Administración avanzada de red desde la línea de comandos.

NETSH (Network Shell) es la herramienta de configuración de red más potente de Windows. Es un shell interactivo o se puede usar con comandos directos. Permite configurar prácticamente todo lo relacionado con la red.

Contextos principales de NETSH

ContextoPara qué sirve

Comando conocido: ver contraseñas WiFi

C:\> netsh wlan show profiles # Lista todas las redes WiFi que guardó tu equipo alguna vez C:\> netsh wlan show profile name="MiRedWiFi" key=clear # Muestra la contraseña en texto plano Tipo de autenticación : WPA2-Personal Tipo de cifrado : CCMP Contenido de clave : M1P4ssword2026 ← ahí está la clave
¿POR QUÉ FUNCIONA?

Windows guarda los perfiles WiFi cifrados en el registro del sistema, pero los descifra para mostrártelos si tenés permisos de administrador. Las claves se almacenan con DPAPI (Data Protection API), vinculadas al usuario del sistema. Por eso este comando requiere CMD como administrador.

NMCLI en Linux

NMCLI es la CLI de NetworkManager, el gestor de red estándar en distribuciones modernas de Linux. Permite gestionar conexiones de forma persistente (sobreviven reinicios).

COMANDO 11

CURL / WGET WIN + LINUX

HTTP desde la terminal: el navegador sin interfaz gráfica.

CURL (Client URL) es una biblioteca y herramienta de transferencia de datos que soporta más de 25 protocolos: HTTP, HTTPS, FTP, SMTP, SFTP, y más. Es indispensable para desarrolladores y para testing de APIs.

WGET es más simple pero muy efectivo para descargas, especialmente con la capacidad de continuar descargas interrumpidas (-c) y hacer mirrors de sitios web (-r).

$ curl -I https://google.com # solo headers HTTP HTTP/2 301 location: https://www.google.com/ content-type: text/html; charset=UTF-8 strict-transport-security: max-age=31536000 ← HSTS habilitado $ curl -X POST https://api.ejemplo.com/users \ -H "Content-Type: application/json" \ -H "Authorization: Bearer TOKEN" \ -d '{"nombre": "Ana", "email": "ana@test.com"}' $ curl https://api.ipify.org # ver mi IP pública 203.0.113.42

Sub-concepto: HTTP y HTTPS

HTTP (HyperText Transfer Protocol) es el protocolo de la web. Define cómo un cliente (navegador) solicita recursos a un servidor y cómo el servidor responde.

Métodos HTTP

MétodoPara qué sirve
GETObtener un recurso (página, datos)
POSTEnviar datos al servidor (crear)
PUTActualizar un recurso completo
PATCHActualizar parcialmente un recurso
DELETEEliminar un recurso
HEADComo GET pero solo headers (sin body)
OPTIONSVer qué métodos acepta el servidor

Códigos de estado HTTP

RangoCategoríaEjemplos
1xxInformacional100 Continue
2xxÉxito200 OK, 201 Created, 204 No Content
3xxRedirección301 Moved, 302 Found, 304 Not Modified
4xxError del cliente400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found
5xxError del servidor500 Internal Error, 502 Bad Gateway, 503 Service Unavailable

HTTPS = HTTP + TLS

HTTPS agrega una capa de cifrado TLS (Transport Layer Security) sobre HTTP. Esto garantiza:

  • Confidencialidad: nadie puede leer el tráfico aunque lo intercepte
  • Integridad: nadie puede modificar los datos en tránsito
  • Autenticación: el certificado SSL/TLS prueba que el servidor es quien dice ser
DATO INTERESANTE

El candado verde en el navegador solo significa que la conexión está cifrada, NO que el sitio es confiable. Un sitio de phishing también puede tener HTTPS con certificado válido. El certificado solo prueba que ese dominio es tuyo, no que tus intenciones son buenas.
